# OSSICRO — Rapid-Access Vision & Expanded Plan (2026-07-09)

_Synthesized from six Fable ideation agents. The rapid-access paperwork engine — AI that turns reference docs + templates into complete, correct, professionally-framed paperwork for single-patient trials / individualized sites / expanded access — is the critical component._

# OSSICRO — Expanded Plan: The Rapid-Access Engine

## 1. The vision, sharpened

OSSICRO exists to make one thing routine: a treating physician, with a patient who has exhausted standard options and a plausible investigational therapy, obtains the complete, correct, professionally-written regulatory paperwork to pursue lawful access — in an afternoon, not a quarter — and knows exactly which counterparties must still say yes.

Two capabilities carry the vision. The first is matching that is genuinely useful: a physician or a patient supplies clinical information through the platform and receives a defensible, citation-backed set of access routes, not a keyword-ranked trial list. The second — the critical one — is a paperwork engine that assembles the full document set for a single-patient expanded-access request, an individual-patient site, or a sponsor-investigator IND directly from OSSICRO's reference templates, with structured hooks into the CRO and pharmaceutical layers, so the request lands on the counterparty's desk in the shape their own review machinery expects.

Two constraints are non-negotiable and define the product as much as the features do. **Correctness:** every fact in every document traces to a source; nothing is invented; every completeness claim is independently verifiable. **Voice:** every document reads as professional regulatory work product, free of LLM mannerisms, because a physician's name is on every page and an obviously-machine-authored filing is a rejection signal to both FDA reviewers and pharma medical-affairs readers.

The hard line does not move. OSSICRO drafts complete documentation for qualified human review. It never performs a non-delegable act — consent, IRB judgment, causality adjudication, statistical sign-off, signature, submission to FDA, or the manufacturer's supply decision. The Claude Agent SDK is confined to four roles — drafting, extraction, adversarial checking, judging — inside deterministic rails (form fill, consistency, clocks, gates, Part 11 signatures). Speed scales with agent parallelism; the eight non-delegable gates never move.

The governing insight across all six facets: **correctness at obscene volume cannot be inspected in after the fact — it must be made structurally impossible upstream.** One canonical fact record that every parallel drafting agent transcribes makes cross-document consistency hold by construction. Provenance triples make invention detectable per span. A register linter makes slop a blocking defect equal to a missing field. The completeness ledger is the computed definition of done: zero reds, every amber a named human's act. The physician's calendar then compresses to answering the ledger's open questions and one triaged review-and-sign sitting, with time-to-drug bounded only by the counterparties the law says must decide.

## 2. The rapid-access paperwork engine (the critical component)

### 2.1 Fact Spine — consistency by construction

A single canonical `StudyFacts` record (PI identity and credentials, IND number, protocol version and date, drug, dose/route/duration, site, IRB of record, sub-investigators, manufacturer) is the only permitted source for any fact appearing in any document. It becomes a first-class engine entity keyed to the 60-document registry's `required_fields`.

N drafting agents run in parallel — one per document in the elected route's registry subset — each contractually bound to two rules: never assert a fact absent from the spine (a missing datum renders as `[[MISSING]]` → red, never an invention), and emit a provenance triple per generated span. Because every document transcribes the same spine, the classic multi-document failure mode (PI name agrees on the 1572 but not the delegation log) cannot occur. The deterministic consistency checker in `check.py` becomes a verification of the invariant rather than the sole defense. This is the mechanism that makes six documents and sixty documents carry the same marginal correctness cost.

The SDK orchestrator fans out one drafting subagent per registry document, each with a bounded context — its template, the spine, and only the reference excerpts it needs — and no shared chat state between drafters. An extraction agent populates the spine from intake artifacts (CV, license, prior 1572, manufacturer package insert), with a human confirming each extracted fact before it enters the spine.

### 2.2 Fill-only-the-blanks generation contract

The drafting agent never emits document prose. It emits only typed field values for the placeholders enumerated in each template's Fields table (which already maps placeholder → study-record path → citation). The engine performs the interpolation into the human-vetted template. Enforcement is a deterministic diff: the rendered document minus the template must reduce to placeholder substitutions only; any other byte delta fails generation. Verbatim-required passages — the 21 CFR 50.25(c) ClinicalTrials.gov statement, the 1572 Field 9 commitments block — are SHA-256-locked in registry entries and hard-fail on any change.

Implementation: parse each template into an AST of `{fixed-prose spans, placeholder slots, [INSTRUCTION] blocks}`. The drafting agent returns structured field values (SDK `messages.parse` with a per-template Pydantic schema generated from the Fields table, `strict: true`). The engine interpolates and runs the template-diff invariant. Instruction blocks are stripped mechanically at finalize, never by the model. This narrows the anti-slop problem from "judge whole AI-written documents" (where LLM judges are unreliable) to "lint and judge short field values" (where a cheap deterministic layer plus a span-scoped checker actually delivers).

### 2.3 Red-Resolver — the ledger-driven intake loop

The completeness ledger already states, for every red, the exact question whose answer resolves it. The Red-Resolver closes the loop. An interview agent walks the physician (or the patient, for patient-side facts) through precisely those questions — and only those — in dependency order (spine facts before the documents that consume them), recomputing the ledger after every answer, until reds hit zero.

Where a red names a chart datum, it becomes a SMART-on-FHIR fetch request through the privacy state machine rather than a question. Where it names a counterparty artifact (manufacturer LOA, IRB concurrence letter), it becomes a tracked outbound request with a drafted letter attached. The session's definition of done is a computed state — ledger: 0 red, every amber mapped to a named human — not a feeling of completeness. Each answer is written into `StudyFacts` provenance-stamped `physician-attested, date, session`. The agent supplies data that makes predicates pass; it never grades, never dispositions a red (a not-applicable call is a human's audited act), never touches amber.

### 2.4 Route-3926 Sprint Pack — the first end-to-end vertical

Individual-patient expanded access (21 CFR 312.310) is the smallest complete instance of the rapid-access story: one physician, one patient, one drug, days not months, treatment not research. It ships first.

From a structured intake the pipeline emits the complete Form 3926 package: the filled 3926 with the 10.a/10.b waiver logic applied, the clinical-history and treatment-plan narrative, the manufacturer LOA-request letter, the IRB chair-concurrence submission, the drug-accountability record shell, and the 50.25-compliant ICF — each artifact carrying provenance triples, the package shipping with its completeness ledger as the cover sheet. An emergency-mode variant starts the 312.310(d) 15-working-day written-follow-up clock and the 56.104(c) 5-working-day IRB-notification clock at telephone authorization, and produces the phone-authorization briefing sheet (what the physician says to FDA on the call).

The requirements universe is small and fixed: 3926 fields per the FDA instructions, 312.305(a)/312.310(a) criteria coverage, the LOA-reference-or-FDA-division-contact rule, the consent-before-treatment gate, and ClinicalTrials.gov suppression per 42 CFR 11.10. Drafting subagents fill the narrative sections strictly from intake data; deterministic code fills form fields and computes the emergency clocks. Gates stay intact: the physician's signature and certifications, the 312.310(a) risk determination, the manufacturer's LOA, IRB concurrence, and the consent event are all ambers that only external humans discharge. Target physician time is at or below 8–15 hours; acceptance test is a synthetic patient fixture reaching a green-or-explained ledger in under 24 hours of wall-clock with only gate actions pending.

### 2.5 Sign-and-Ship Cockpit — verifiable assembly and single-pass gate discharge

The terminal stage the pipeline implies but does not yet ship. It takes a ledger-green/amber package and produces: (1) a hash-chained package manifest — every artifact, version, provenance index, and ledger snapshot, independently checkable by an IRB or pharma counterparty; (2) the single-pass review surface, with deterministic and boilerplate spans collapsed (provenance on demand) and inferred/interpretive spans foregrounded, the amber list rendered as an ordered signature-routing queue naming each act, its human, its authority, and its evidence-of-discharge; (3) Part 11 sign-off capture (§§11.50–11.300) that discharges ambers one by one and flips document states; (4) final export — filled 3926/1571/1572 PDFs on the official forms, the IRB bundle, the manufacturer packet — unlocked only when the state machine permits, with submission itself remaining an explicit human-authorized act.

Official-form PDF fill is deterministic AcroForm field mapping (the 3926/1571/1572 schemas are fixed and published). No SDK agent touches this stage except an optional cover-memo drafter, itself provenance-bound and lint-checked. `GateViolation` remains the enforcement: export before discharge raises, correctly.

### 2.6 Golden-Package Harness — keeping correctness true at volume

A regression harness of synthetic study fixtures (emergency 3926, non-emergency 3926, Route-2 site dossier, Route-3 full IND, pediatric-assent ICF, non-English short-form set, no-LOA-possible branch), each paired with a human-verified gold package. Every change to a template, rule, prompt, or model version replays the full set and reports: form-field exactness, provenance coverage, ledger-grade agreement with gold, register-lint pass rate, and — the red line — zero gate regressions. This is the engine's own credibility file under the FDA 2025 AI-credibility draft guidance: a measured, versioned record that the drafting context-of-use performs as claimed, which is what lets OSSICRO defend a low-influence/low-consequence COU classification with data rather than assertion.

## 3. Patient intake and matching (Claude Agent SDK)

### 3.1 Predicate-driven minimum-necessary intake

Intake is not a fixed form. It is computed backward from the candidate trials' parsed eligibility predicates. Phase one collects the small early-phase eligibility kernel — age/sex, primary diagnosis and histology, stage, prior lines of therapy, ECOG, the core organ-function labs (ANC, platelets, Hgb, creatinine/CrCl, bilirubin, AST/ALT), biomarker/variant status, major exclusion comorbidities, current meds, pregnancy/contraception — just enough to seed recall-first retrieval. Phase two lets the adjudication stage's `indeterminate — needs data` verdicts define, per candidate, the exact additional questions worth asking.

A gap agent diffs the predicates referenced by live candidates against the fields present in the profile and emits an ordered question list, ranked by how many candidates each answer resolves. The loop terminates when no live candidate has an unresolved, patient-supplyable indeterminate criterion. This makes minimum-necessary (45 CFR 164.502(b)) verifiable from the audit trail — every asked datum traces to a specific parsed predicate — rather than merely asserted. Extraction agents never guess an absent datum: an unmappable field becomes `indeterminate`, never a silent not-met, preserving the recall-first safety asymmetry from intake through adjudication.

### 3.2 Multi-source extraction

A Claude SDK extraction agent ingests heterogeneous inputs — a pasted chart snippet, free-text history, a dictated summary, a PDF/NGS report path, copy-pasted lab panels — and emits a structured profile mapped to US Core / mCODE / Genomics Reporting profiles, with span-level provenance and per-field confidence. Anything it cannot map to a coded value with confidence is emitted as `indeterminate`, never guessed. This is the on-ramp for the solo or community clinician whose usable data arrives as text, not a clean FHIR bundle. It runs strictly inside the PREPARATORY privacy state (local, no PHI egress, ephemeral, audited reads) and never writes back to any EHR. The extracted profile is a draft the physician confirms field-by-field, with deterministic and inferred spans visually segregated, before it drives matching.

A specialized genomic path normalizes NGS panels and molecular-pathology reports to HGNC gene and HGVS variant nomenclature, structures them as FHIR Genomics / mCODE observations, grades therapeutic relevance against OncoKB (FDA-recognized levels) and CIViC as labeled edges, and feeds the biomarker→pathway→target→trials mechanism-expansion graph. This is where the largest recall win lives: a colorectal patient with BRAF V600E surfaced into a BRAF-mutant basket trial not indexed under "colorectal." Interpreting a variant for therapeutic action remains the practice of medicine; the agent surfaces cited candidates only.

### 3.3 Rapid-access route classifier — the bridge to paperwork

The moment matching stops being data collection and starts making the physician able to act. A triage agent takes the structured profile and the adjudicated candidate ledger and drafts a rapid-access route recommendation across the pathways — enroll in an existing trial (Mode A), stand up a sponsor-investigator n-of-1 (Mode B), pursue expanded access (312.310 / Form 3926), or a designed "no suitable option" outcome — with the hard-stop gates surfaced up front, before any dossier drafting: central-IRB access and fee, research-liability indemnification, Medicare/NCD 310.1 billing qualification, and physical site-readiness scored against the protocol's schedule of assessments.

This is where a physician who would otherwise discover the insurance gap after assembling the package is instead told at triage. Output is a draft recommendation with each gate as a green/amber/red ledger item, never an instruction to proceed. The pathway decision, the therapeutic-misconception judgment, and "is Mode B ethical for this patient" stay with the physician and IRB.

### 3.4 Dual front door and confirmed-profile handoff

Two intake entry points share one extraction/matching core under different egress rules. The physician path wires SMART App Launch (point of care, OAuth2+PKCE) or Backend Services bulk `$export` (panel screen, signed-JWT), read-only scopes, no PHI egress. The patient self-intake path runs the free-text extraction agent locally, produces a de-identified safe-harbor seed (164.514(b)) plus a physician-invite artifact, and withholds the raw candidate list from the patient portal by design — replaced with the transparency-floor status view and stated recourse. Self-intake never enrolls anyone.

On profile confirmation the structured profile is hashed as the generation input-of-record. Every confirmed `met` criterion and its datum citation pre-populates the screening/eligibility worksheet and downstream activation documents as drafts; every unresolved `indeterminate` becomes an amber ledger item carrying its resolving question. Intake data is entered once and flows into the paperwork — no re-keying the same diagnosis and labs into the eligibility worksheet, the 1572 support, and the protocol synopsis. The input-hash makes every generated draft reproducible and tamper-evident (Part 11 / ALCOA++). This is what makes "genuinely useful matching" continuous with "complete paperwork built rapidly."

## 4. CRO and pharmaceutical hooks

The integration surface is not an API. Sponsors will never grant write access to validated CTMS/EDC systems, and their reviewers will never execute third-party software. The surface is therefore document-shaped and contract-shaped: packages rendered in the recipient's own conventions and checklists, joined to signable instruments that legally constitute the connection. Standardize the package, not the pipe.

### 4.1 Say-Yes packages — route-specific, sponsor-native exports

A one-click export layer that renders a physician's package in exactly the shape the receiving pharma function consumes. Three variants keyed to the route triage: the expanded-access LOA-request pack (3926 draft, clinical narrative, monitoring plan, physician-qualification statement, consent form, IRB path, signature-ready LOA skeleton); the IIS pack (concept synopsis, protocol draft, qualification dossier, FMV-benchmarked budget, support-agreement and SDEA scaffolds shaped for a Medical Affairs review committee); and the full activation binder in TMF Reference Model naming and indexing. Every pack ships with a professional cover letter, a human-readable cross-reference index (requirement → artifact → citation → signer), visible-signature PDF renditions, and the hash manifest as an appendix nobody is required to run. The completeness ledger blocks export while any gate item is amber or red; nothing leaves without the physician's explicit release action.

### 4.2 Manufacturer Front-Door Registry and route router

A maintained, machine-readable directory of every manufacturer's sanctioned intake points — the §561A-mandated published expanded-access policies, IIS/grants portals, and medical-information contacts — normalized into structured records (counterparty function, required intake fields, submission format, drug-specific programs, stated decision timelines). The router takes the drug facts, applies the route-likelihood ordering, resolves the drug to its manufacturer's actual front door, and pre-maps the Say-Yes package fields onto that company's specific intake form. Hard rail: routing to a sales contact is a validation error, never a fallback — always Medical Affairs or Clinical Development site-selection. An extraction agent crawls published policies into a versioned registry (source URL, retrieval date, hash per record); a freshness agent re-checks on a schedule and diffs; every entry that will drive an actual submission is human-confirmed before first use, because a stale contact wastes the patient's clock. The registry is itself an open-source public good.

### 4.3 Hooks-as-Contracts — LOA, SDEA, TORO builders

The three signable instruments that legally constitute the connection, built as first-class engine documents. The LOA skeleton is the §312.305(b)(1) cross-reference letter drafted for the manufacturer's counsel to edit and sign, minimizing their drafting cost to near zero. The SDEA scaffold wires bidirectional ICSR flow with the 312.32 7/15-day clocks reconciled against contractual reporting to the manufacturer's pharmacovigilance. The TORO builder forces obligation-by-obligation enumeration across the full 312.50–312.59 transferable set — every obligation explicitly marked TRANSFERRED or RETAINED (encoding 312.52's silence-means-retained rule as a completeness check), each transferred item bound to a named qualified human at the micro-CRO, and non-transferable investigator duties (312.60–312.69) structurally excluded from the picker. A TORO-specific validation rule fails the check pass if any obligation lacks an explicit mark, if a transferred obligation lacks a named qualified human, or if any 312.60–312.69 item appears.

### 4.4 Checklist Join — sponsor-questionnaire ingestion

Pharma will never adopt OSSICRO's format; OSSICRO must adopt theirs. This tool ingests the sponsor's own artifact — feasibility questionnaire, site-qualification checklist, IIS application, EA intake form (PDF/DOCX/XLSX/portal fields) — and generates the join: each sponsor line-item mapped to the specific OSSICRO artifact, governing citation, and signer, with unmet items rendered as an explicit gap list carrying the exact resolving question, in the sponsor's own ordering and vocabulary. Their reviewer performs a single checklist join inside their ordinary workflow. A matching agent proposes mappings with per-mapping confidence and citation; a human confirms every mapping before the index is generated (a wrong mapping in a diligence response is a credibility-destroying event). Confirmed mappings persist as reusable per-sponsor profiles.

### 4.5 Micro-CRO Workbench and Safety Exchange Conduit

The accountable layer's operating console: a Tier-2 gate queue (packages awaiting qualified-human review, each with the checking agent's pre-review findings attached so human time goes to judgment); a TORO-scoped obligation board (per-study view of exactly which obligations this entity has assumed, each bound to its named human, safety-clock escalations routed to whoever holds 312.32 operations); and an E6(R3) oversight-evidence log (append-only record of every human review act, which doubles as the BIMO inspection-readiness file and the accumulating performance record that wins the second trial). No agent can advance a document past a Tier-2 gate; the queue is structurally a human bottleneck by design.

The Safety Exchange Conduit is the ongoing hook after the yes: adverse events intaken on the physician side are coded (draft MedDRA), narrated, and rendered simultaneously as a MedWatch 3500A draft for the sponsor-investigator's own 312.32 filing and an E2B(R3)-conformant ICSR the manufacturer's safety database (Argus, ArisGlobal) ingests without transformation. Both clocks are computed and escalated by the safety-clock engine. Causality and expectedness determinations, and every transmission, remain human.

## 5. Writing-quality layer (anti-AI-slop)

Voice quality becomes guaranteeable the moment it stops being a property of generation and becomes a property of architecture. Because the templates already separate human-vetted regulatory prose from typed placeholders with per-field citations, the engine confines the model to filling blanks (§2.2), which shrinks the problem to linting and judging short field values. Four layers, cheapest first.

### 5.1 Deterministic register linter with a versioned banned-construction registry

A zero-LLM lint pass in `check.py` driven by `registry/style-rules.json`. A style failure is a check finding like any completeness finding, recorded in the ledger with provenance, and runs in milliseconds on every generation. Each rule carries severity, a confirmed/interpretive classification, and a citation where one exists. Rules resting on interpretive positions fail toward the human gate, never auto-pass.

**Banned constructions (initial registry).** Filler openers and closers: "it is important to note", "it's worth noting", "it should be noted", "in conclusion", "in summary", "at the end of the day", "in today's landscape". Antithesis tics: "not just X, but Y", "it's not only … it's …", "more than just". Inflated verbs and adjectives: "delve", "leverage" (as verb-filler), "seamless", "robust" (as filler), "crucial/vital/pivotal role", "plays a key role", "unlock", "harness", "navigate the complexities", "tapestry", "testament to". Promotional and superlative claims, especially in consent and counterparty-facing text: "promising", "breakthrough", "cutting-edge", "state-of-the-art", "revolutionary", "will improve", "is proven to". Hedging stacks: "may potentially", "could possibly", "might suggest that it may". Structural tic metrics: em-dash density per 1,000 words above threshold, sentence-initial "Additionally/Moreover/Furthermore" rate, three-item-list cadence overuse, rhetorical-question openers. Register-specific rules keyed to document category: consent field values must be second person and non-exculpatory (waiver/release constructions — "you waive", "you release", "the sponsor shall not be liable", "you give up the right to" — are severity-critical, cited to 21 CFR 50.20); FDA-form worksheets must be third person with no first-person editorializing; correspondence must be formal and non-speculative.

### 5.2 Voice-validation checking agent with per-category rubrics

A Claude SDK checking agent that judges only the filled field values (short spans, thanks to §2.2) against an explicit rubric per document category: consent register (plain language, second person, risks stated without minimization, no promotional framing of benefits, payment never listed as a benefit), FDA-form register (factual, transcribable, no narrative), protocol/M11 register (precise, cited, non-promotional), correspondence register (formal, non-speculative, comparable to a reference regulatory document). It returns structured findings — `{field, span, rule_id, severity, rationale, suggested_rewrite}` — and never touches verbatim-locked or fixed template prose, which are out of its jurisdiction by construction. Rubrics live in `registry/voice-rubrics.json` keyed to the template's category. The agent proposes severity; only the deterministic layer or a human can hard-fail a document — the LLM judge alone fails toward the human gate.

### 5.3 Flag-and-rewrite loop

A bounded repair loop that runs after generation and before the human-review gate. Deterministic lint plus the voice agent flag findings → a rewrite agent produces corrected text for the flagged spans only (it receives the span, the finding, and the field's citation and rubric — not license to rewrite the document) → the deterministic lint, verbatim-hash check, and template-diff invariant re-run on the result → loop until clean or three iterations, then surface any residual findings to the human alongside the draft. Rewrites are span-scoped; the engine splices them; the model never returns whole documents; verbatim-locked and fixed-template text can never be a rewrite target. Every rewrite is recorded as a provenance event (original text, finding, replacement, agent, timestamp), so the audit trail shows what the AI changed and why. What lands in the physician's queue already reads as professional regulatory work product; their time goes to substance, not copy-editing.

### 5.4 Adversarial QC lane and golden-corpus regression

An independent Claude QC agent with zero shared context with the drafters red-teams every package for internal contradictions, unsupported spans, stale versions, and missed conditional requirements. It cannot edit documents; it files findings as reds (fix required) or annotations on ambers, preserving the drafter/checker structural separation. The human reviewer receives draft plus machine critique together, so review starts where the adversary found weakness.

The controlling failure mode of any anti-slop system is flagging legitimate regulatory boilerplate as slop — regulatory register legitimately uses "shall", formal passives, and defined-term repetition that naive lists punish. A golden-corpus regression harness prevents this: a golden set of genuine regulatory prose (FDA model consent language, the 1572 commitments, M11 exemplar sections, passages from the source library) that must pass with zero findings, and a slop set of the same passages deliberately degraded with mannerisms that must be flagged, with measured per-rule precision/recall run in CI on every change to the rule pack, the rubrics, or the model version. CI target: zero findings on golden (hard), ≥95% of injected mannerisms flagged on slop (tracked, tunable). This makes "the paperwork reads professionally" a tested claim rather than an asserted one, and lets the banned-construction registry evolve without eroding physician trust.

## 6. Per-persona value and adoption

Each persona's "yes" is gated by a different scarce currency, and none of them is "more documents."

**Physician (the load-bearing persona).** The blocker is not any single document — it is not knowing the total burden and the hidden hard stops before committing. The adoption trigger is the **One-Visit Feasibility Verdict**: enter the clinical picture and target drug, receive in one sitting the route triage, the four software-unfixable hard stops (central-IRB access and fee, research-liability gap, NCD 310.1 billing pre-screen, site-readiness against the schedule of assessments) named as the *first* output, an honest hours/cost table for the chosen route (~8–15h expanded access, ~25–40h Mode A, ~80–150h Mode B), and the complete draft dossier if they proceed. The **IND-Maintenance Copilot** then converts year two into a form of the day-one experience — driving the safety clocks, drafting DSURs and amendments, tracking continuing-review dates — cutting the standing 10–20h/month maintenance burden to its irreducible judgment core, which is what turns OSSICRO from a startup tool into an ongoing practice tool.

**Patient.** The barrier is often the absence of a physician with a research office. The **Story-to-Clinical-Picture intake agent** gives them a genuinely useful action they can complete alone — narrate the situation, upload records — producing a de-identified structured profile that lands where it is clinically actionable, with the physician-mediation gate preserved. The **plain-language status portal**, generated from the completeness ledger against a pre-approved IRB-reviewable string library, ensures they are never left watching a spinner: "your doctor received N candidate programs on [date]", expected-response windows, honest "no suitable option found" framing, no string that implies enrollment likelihood pre-IRB-approval. Expanded access is the purest rapid-access wedge — treatment, not research — and the fastest lawful route to drug.

**Micro-CRO.** A thin entity with 312.52(b) enforcement parity can only stay thin if its per-obligation overhead is near zero. **Micro-CRO-in-a-Box** generates the quality-system substrate (the enumerated minimum SOP set, training-record scaffolding, a recurring BIMO-readiness self-audit that files findings as draft CAPAs), the **TORO builder** scopes obligations lawfully and refuses to draft an unlawful scope, and the **Tier-2 Review Workbench** concentrates one qualified reviewer's attention on judgment calls so throughput multiplies at equal or better quality — the difference between a viable escalation business and one that cannot scale past its founder.

**Pharma.** The trust gate is entity-shaped and evidence-shaped: verification must be cheaper than rejection. The **sponsor-side dossier verifier** — a free, standalone, no-install tool that confirms hash-chain integrity, checklist coverage, internal consistency, and open items without trusting OSSICRO's servers — turns "trust this new one-physician site" into "verify this dossier in twenty minutes." **Pharma-shaped submission export** (IIS portals, EA intake per the manufacturer's published §561A policy, the SDEA draft) delivers requests in the shape Medical Affairs expects, respecting the MA/CD firewall by construction. Every accountable act still terminates in a qualified human per the non-delegable gate list.

## 7. Build roadmap

The existing dependency chain (wiki → templates → engine → coordination → matching → entity) is preserved. The rapid-access vision re-weights sequencing so the first *end-to-end vertical a physician can actually use* ships as early as correctness allows, rather than after the full IND engine. The standing rule holds across every phase: no phase introduces an automated path through any gate in the master gating matrix; each phase's adversarial review explicitly attempts to find one, and finding one is a release blocker.

**Phase 2.5 — Fact Spine + anti-slop substrate (insert after the 1572 loop).** Formalize `StudyFacts` as the engine's canonical fact entity. Land the deterministic register linter (§5.1) and the golden-corpus harness (§5.4) now, because retrofitting voice enforcement after documents multiply is structurally dishonest, and because the linter is the cheapest correctness layer in the system. Success criterion: a generated 1572 passes the template-diff invariant and the register lint with zero findings on the golden corpus, and any injected mannerism is caught.

**Phase 3-EA — Route-3926 Sprint Pack (elevate ahead of the full IND engine).** Ship the smallest complete rapid-access vertical (§2.4): the expanded-access package end-to-end, the Red-Resolver loop (§2.3), the voice agent and flag-and-rewrite loop (§5.2–5.3), and the Sign-and-Ship Cockpit for this one route (§2.5). Success criterion: a synthetic patient fixture goes intake-to-green-ledger in under 24 hours of wall-clock with only the physician's gate actions (sign, consent, IRB submission) and the manufacturer's supply decision pending, and every field traces to its source.

**Phase 3-IND — IND assembly with the Fact Spine swarm.** The full 312.23 Mode B package generated by the parallel drafting swarm (§2.1) with the adversarial QC lane and cross-document consistency engine (§5.4). Success criterion: a complete, internally consistent draft IND with a fully green-or-explained ledger, PI/site/IND-number/protocol-version identity verified across every document.

**Phase 4 — Coordination, safety-clock engine, and counterparty conduit.** Inter-entity routing, the milestone tracker, the dedicated safety-clock engine computing 7/15-day deadlines (compute and escalate only; causality and filing stay human), and the Safety Exchange Conduit's E2B(R3) renderer (§4.5). The communication hub ships with the Medical-Affairs/Clinical-Development firewall. Success criterion: a simulated study in which every document reaches every entitled counterparty, every deadline fires with correct lead time, and no gated act has an automated path.

**Phase 5 — Matching, route classifier, and the dual front door.** The retrieve→adjudicate matching layer (§3.1) with mechanism-aware expansion and three-valued verdicts, the multi-source and genomic extraction agents (§3.2), the rapid-access route classifier (§3.3), the dual front door and confirmed-profile handoff (§3.4), all behind the code-enforced privacy state machine, with the evaluation harness run on CHIA/TREC/n2c2 *before* any patient-facing claim of matching quality. Single-container local deployment ships with it. Success criterion: benchmark recall/precision published; the privacy state machine demonstrated to block PHI egress and hard-gate the preparatory→enrollment transition; eligibility verdicts carry citations an investigator can check.
